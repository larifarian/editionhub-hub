HASE → CIDOC-CRM

Wie HASE intern auf CIDOC-CRM abgebildet wird — das internationale Referenzmodell für Kulturgut-Dokumentation

Warum CIDOC-CRM?

CIDOC-CRM (ISO 21127) ist die internationale Ontologie für Kulturgut-Dokumentation. EditionHub speichert alle Daten intern als CIDOC-CRM-konformes RDF. HASE ist die einfache Lieferschicht darüber — Projekte müssen weder RDF noch CIDOC-CRM verstehen.

Diese Seite dokumentiert das Mapping für alle, die es genau wissen wollen: RDF4J-Entwickler, Linked-Data-Experten, oder Projekte, die ihre Daten direkt als RDF liefern möchten.

Entitäten-URIs folgen dem Muster http://id.editionhub.org/entity/{typ}/{projekt}/{id}.

Activity → E7_Activity

Jede HASE-Activity wird zu einer crm:E7_Activity. Der Typ wird über crm:P2_has_type zugewiesen. 

# HASE-JSON:
# { "id": "gb-10280-send", "type": "LETTER_SENDING",
#   "date": "1774-08-16",
#   "actor": { "type": "person", "label": "Goethe", "gnd": "118540238" },
#   "place": { "label": "Frankfurt am Main", "gnd": "4018118-2" } }

# Wird zu RDF:
<http://id.editionhub.org/entity/letter-sending/goethe-biographica/gb-10280-send>
    a                          crm:E7_Activity ;
    crm:P2_has_type            eh:LETTER_SENDING ;
    crm:P175i_starts_after_or_with_the_start_of
                               <http://id.editionhub.org/entity/tag/1774-08-16> ;
    crm:P184_ends_before_or_with_the_end_of
                               <http://id.editionhub.org/entity/tag/1774-08-16> ;
    crm:P14_carried_out_by     <https://d-nb.info/gnd/118540238> ;
    crm:P7_took_place_at       <https://d-nb.info/gnd/4018118-2> .

URI-Strategie: Personen und Orte mit GND-ID verwenden direkt die GND-URI (https://d-nb.info/gnd/...). Ohne Normdaten wird eine projektlokale URI erzeugt und per owl:sameAs mit dem EditionHub-Index verknüpft.

Datum → Tag-URI

Datierungen werden über Tag-URIs modelliert. Die CRM-Properties P175i (starts after or with) und P184 (ends before or with) ermöglichen exakte und unscharfe Zeitangaben:

HASEP175i (von)P184 (bis)
"1774-08-16"tag/1774-08-16tag/1774-08-16
qualifier: "earliest"tag/{value}— (offen)
qualifier: "latest"— (offen)tag/{value}
{ from, to }tag/{from}tag/{to}
 
# Exaktes Datum: "1774-08-16"
<...activity-uri>
    crm:P175i_starts_after_or_with_the_start_of  <.../tag/1774-08-16> ;
    crm:P184_ends_before_or_with_the_end_of      <.../tag/1774-08-16> .

# Spätestens: { value: "1774-08-20", qualifier: "latest" }
<...activity-uri>
    crm:P184_ends_before_or_with_the_end_of      <.../tag/1774-08-20> .

# Zeitspanne: { from: "1786-09-11", to: "1786-09-12" }
<...activity-uri>
    crm:P175i_starts_after_or_with_the_start_of  <.../tag/1786-09-11> ;
    crm:P184_ends_before_or_with_the_end_of      <.../tag/1786-09-12> .

Thing → E22_Human-Made_Object

Das historische Objekt einer Handlung (z.B. der Brief bei einer Briefsendung) wird als crm:E22_Human-Made_Object modelliert und über crm:P16_used_specific_object mit der Activity verknüpft. 

# thing: { sourceRef: "z-gsa-28-5" }
<...activity-uri>
    crm:P16_used_specific_object  <.../brief/goethe-biographica/z-gsa-28-5> .

<.../brief/goethe-biographica/z-gsa-28-5>
    a                  crm:E22_Human-Made_Object ;
    crm:P2_has_type    eh:LETTER .

# thing.references → P67_refers_to (inhaltliche Bezüge)
<.../brief/goethe-biographica/z-gsa-28-5>
    crm:P67_refers_to  <https://d-nb.info/gnd/118722026> ;  # Kestner
    crm:P67_refers_to  <https://d-nb.info/gnd/4099312-4> .  # Werther

Source → E31_Document

Quellen werden als crm:E31_Document modelliert. Die Verbindung zur Activity läuft über crm:P70_documents. 

# source: { id: "z-gsa-28-5", type: "LETTER",
#   archive: { label: "GSA", gnd: "5080444-3" },
#   shelfmark: "GSA 28/5",
#   attestsActivity: ["gb-10280-send", "gb-10280-receive"] }

<.../zeugnis/goethe-biographica/z-gsa-28-5>
    a                  crm:E31_Document ;
    crm:P2_has_type    eh:LETTER ;
    crm:P70_documents  <.../letter-sending/goethe-biographica/gb-10280-send> ;
    crm:P70_documents  <.../letter-receiving/goethe-biographica/gb-10280-receive> ;
    crm:P67_refers_to  <https://d-nb.info/gnd/118722026> ;
    crm:P67_refers_to  <https://d-nb.info/gnd/4099312-4> ;
    eh:archive         "Goethe- und Schiller-Archiv" ;
    eh:shelfmark       "GSA 28/5" ;
    eh:editionUrl      "https://goethe-biographica.de/briefe/gb-10280" ;
    eh:catalogUrl      "https://kalliope-verbund.info/DE-611-HS-1234567" .

Context → E5_Event

Zusammenhänge (Korrespondenz, Begegnung, Reise) werden als crm:E5_Event modelliert. Die zugehörigen Activities werden per crm:P9_consists_of eingehängt. 

# context: { id: "gb-10280-korr", type: "CORRESPONDENCE" }

<.../korrespondenz/goethe-biographica/gb-10280-korr>
    a                   crm:E5_Event ;
    crm:P2_has_type     eh:CORRESPONDENCE ;
    crm:P9_consists_of  <.../letter-sending/.../gb-10280-send> ;
    crm:P9_consists_of  <.../letter-receiving/.../gb-10280-receive> .

Verkettung → P134

Handlungsketten (z.B. Briefsendung → Briefempfang) werden über die CIDOC-CRM-Properties P134_continued und P134i_was_continued_by abgebildet. 

# gb-10280-send → wasContinuedBy → gb-10280-receive

<.../letter-sending/.../gb-10280-send>
    crm:P134i_was_continued_by  <.../letter-receiving/.../gb-10280-receive> .

<.../letter-receiving/.../gb-10280-receive>
    crm:P134_continued          <.../letter-sending/.../gb-10280-send> .

Normdaten-Auflösung

Beim Ingest löst EditionHub GND-IDs gegen den internen Index auf. Bekannte Entitäten werden per owl:sameAs verknüpft, neue Entitäten werden zur Aufnahme vorgeschlagen. 

# Person mit GND → direkte GND-URI
<.../activity-uri>
    crm:P14_carried_out_by  <https://d-nb.info/gnd/118540238> .

# Im EditionHub-Index existiert:
<http://id.editionhub.org/entity/person/goethe>
    owl:sameAs  <https://d-nb.info/gnd/118540238> ;
    rdfs:label  "Goethe, Johann Wolfgang von" .

# Person ohne GND → projektlokale URI
<.../activity-uri>
    crm:P14_carried_out_by  <http://id.editionhub.org/entity/person/goethe-biographica/mueller> .

Named Graphs

Jeder HASE-Import erzeugt einen Named Graph in RDF4J. Das ermöglicht versionierte Importe — ein neuer Import ersetzt den vorherigen Graph. 

# Graph-URI: http://id.editionhub.org/graph/{projekt}/{datum}

<http://id.editionhub.org/graph/goethe-biographica/2026-04-06>
    rdfs:label     "Goethe Biographica" ;
    skos:notation  "goethe-biographica" .
RDF4J RepositoryInhalt
editionhub-dataActivities, Sources, Objects — pro Projekt ein Named Graph
editionhub-indexPersonen, Orte, Handlungstypen — projektübergreifend

eh: Erweiterungen

Für Felder, die CIDOC-CRM nicht direkt abdeckt, verwendet EditionHub den eigenen Namespace http://id.editionhub.org/ontology#. Die vollständige OWL-Ontologie ist unter editionhub.org/hase/ontology abrufbar.
